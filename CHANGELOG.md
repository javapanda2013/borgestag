## [1.46.41] - 2026-06-04

### Performance — 別ウィンドウ自動同期の再描画を差分化し、操作反映後のメモリ増大を解消

保存履歴の件数が多い環境で、お気に入りクリックなどの操作の反映後にブラウザのメモリ使用量が数 GB 単位（最大 8GB 近く）まで増大する問題を解消しました。Firefox プロファイラでの計測で、別ウィンドウの自動同期が、1 件の変更通知のたびに保存履歴の全件をウィンドウ間でコピー（プロセス間転送）していたことが主因（最大要因の割当）と特定しました。

- 別ウィンドウの自動同期を「変更通知のたびに保存履歴の全件を取得・転送」から「変更／追加された項目だけを 1 件取得して差し込む差分方式」へ変更しました。これにより 1 操作あたりの転送量が件数に比例しなくなります。
- 取り込み・移送などの一括処理時のみ、従来どおり全件取得にフォールバックします（移送前の状態でも安全側で全件取得）。
- 表示の並び順・絞り込み・ページングの挙動は従来と同一です（差分後の配列を保存日時の降順に整列し、全件取得時と同一の並びを保証）。

[design-doc-skip: 設計書反映済 — D_IndexedDB_スキーマ.md §4.3（cross-context refresh の差分化 GET_HISTORY_ENTRY / _patchSaveHistory 配列 patch）、07_事故事例 §15。書込差分化（v1.46.40）に続く「変更通知後の再取得」差分化。設計書反映 gate は拡張機能リポジトリの diff のみ検出のため marker で auditable 化]

## [1.46.40] - 2026-06-04

### Performance — 保存履歴の各操作を差分更新化し、操作時の重さ・フリーズを解消

保存履歴の件数が多い環境で、お気に入りのクリックや削除・グループ化・タグ編集などの操作のたびに動作が重くなり、フリーズすることがある問題を根本的に解消しました。原因は、1 件の変更でも保存履歴の全件を読み込み・全件書き込みしていたためです。

- 保存履歴の更新を「全件の読み込み＋全件の書き込み」から「変更した項目だけを更新する差分方式」へ全面的に変更しました。対象：お気に入り（単体・一括）、削除（単体・複数・期間）、グループ化／解除、タグ・権利者の追加／削除／置換／改名、編集パネルの保存、画像保存、サムネイル生成。
- 別ウィンドウの自動同期は、変更箇所だけを再描画する方式を維持しています。
- データ保護（移送後の空書き込み防止・読み込み失敗時の安全停止）は従来どおり全操作で維持しています（差分方式は全消去を行わないため、構造的にもより安全）。
- あわせて、変更通知の方式を軽量化（全件の文字列化を廃止）し、容量表示を保存履歴の件数表示へ変更しました。

[design-doc-skip: 設計書反映済 — D_IndexedDB_スキーマ.md §4.3（差分更新 _patchSaveHistory）、07_事故事例 §15。全 per-click/編集操作を差分更新化、import/migration は全置換維持（guard 付き）。設計書反映 gate は拡張機能リポジトリの diff のみ検出のため marker で auditable 化]

## [1.46.39] - 2026-06-04

### Fixed — 画像保存などの操作で保存ウィンドウがフリーズする問題（v1.46.38 の回帰）

v1.46.38 の変更で、画像保存などの操作のたびに保存ウィンドウが数秒フリーズする回帰が発生していました（保存件数が多いほど顕著）。原因は、変更箇所が指定されない操作で「保存履歴の全タイルを再描画」していたためで、Firefox プロファイラで数秒の長時間タスクとして確認しました。

- 画像保存時に「追加された 1 件のみ」を通知するよう修正し、保存ウィンドウは該当タイルの追加だけで済むようにしました（全再描画を回避）。
- 別ウィンドウ同期での再描画を、変更箇所が不明な場合も「全再描画」ではなく既存タイルを再利用する方式に変更しました（追加・削除・サムネイル変更は自動反映、内容のみの変更は変更箇所通知で反映）。

## [1.46.38] - 2026-06-03

### Performance — 保存履歴操作時のメモリ使用量を大幅削減

保存履歴のグループ化・削除・編集などの操作のたびに、ブラウザのメモリ使用量が数 GB 単位で増加する問題を解消しました（保存件数が多いほど顕著）。Firefox プロファイラでの計測で、操作ごとに保存履歴の全件を文字列化して変更箇所を検出する処理が主因（累積で数 GB の割当）と特定し、次の見直しを行いました。

- 変更通知を「全件を比較して変化を探す」方式から「操作した側が変更箇所を伝える」方式へ変更。保存件数に比例した重い文字列化処理を廃止しました。別ウィンドウの自動同期は維持し、変更箇所が不明な場合は安全に全体を再描画します。
- 容量表示を、保存領域全体を毎回読み込んで概算する方式から、保存履歴の件数表示へ変更しました（移送後は実体を反映しない指標だったため是正）。保存履歴のバックアップ（復旧用）はそのまま保持しています。

データの保存内容・復旧用バックアップには一切変更はありません（表示と内部処理の最適化のみ）。

## [1.46.37] - 2026-06-02

### Fixed — グループ化等での保存履歴全消失の恒久対策（二段防御）

保存履歴を IndexedDB 専用に移送した後、IndexedDB の読込が一時的に失敗した際に保存履歴が空として扱われ、その空データが書き戻されて全件消失する不具合（過去に発生した同型事象の再発）に対し、構造的な二段防御を追加しました。

- 読込側：移送後に IndexedDB 読込が失敗し、かつ代替の保存先にもデータが無い場合は、空として処理を続行せず中断するようにしました（誤った空データの書き戻しを防止）。
- 書込側：保存履歴を IndexedDB へ書き込む際、既存データが存在するのに空で上書きしようとした場合は、明示的な全件削除を除いて書き込みを中断するようにしました。
- 上記を保存ウィンドウ・設定画面・バックグラウンドの 3 経路すべてに適用しました。

データが保持されている状態で読込に失敗した場合は、保存履歴タブで「読込に失敗しました。再読込してください（データは保持されています）」と表示し、空表示による誤解を防ぎます。

[design-doc-skip: 設計書反映済 — D_IndexedDB_スキーマ.md §4.4・事故 mapping、01_設定画面_保存履歴タブ §5.2/5.3/7.2、07_事故事例 §14 に二段防御を記載。設計書反映 gate は拡張機能リポジトリの diff のみ検出し親リポジトリの設計書更新を機械検知できないため marker で auditable 化]

## [1.46.35] - 2026-05-29

### Changed — ファイル名取得モード ON 時はフォルダ別タグ・手動タグを無視（ファイル名由来のみ付与）

「ファイル名からタグ・権利者を取得する」モードが ON のとき、フォルダ別タグ設定・手動追加タグ・（1枚ずつの）入力タグをすべて無視し、ファイル名から復元したタグ・権利者のみを保存履歴へ付与するようにしました（v1.46.34 ではこれらを併用＝和集合していました）。フォルダ別タグ設定は BorgesTag 導入以前のメタ無しファイルを取り込む用途のもので、BorgesTag が保存したファイル（ファイル名にメタを埋め込み済み）の復旧では不要なため、余分なタグの混入を避ける狙いです。

あわせて、このモードが ON のときは無視されるタグ設定 UI を視覚的に無効化（グレーアウト）し、折りたたんで「無視されます」と明示するようにしました。一括取り込みではフォルダ別タグ設定セクション、1枚ずつ取り込みではタグ編集エリア（メイン・サブ・権利者）が対象です。保存先の入力や保存ボタンはそのまま操作できます。

また、このモードでの取り込みでは、復元したタグを取り込み元フォルダに「保存先」として関連付けないようにしました（取り込み元は復旧の読み取り元であり、保存先ではないため）。

[design-doc-skip: 既存 storage key・UI（extImportFromFilename / ext-from-filename, v1.46.34 で追加済）の挙動調整。新規 key・permissions・データ仕様の変更なし。設計書 画面別/12 に Q-114-1 改修を反映済み]

## [1.46.34] - 2026-05-29

### Added — 外部取込でファイル名からタグ・権利者を取得するモード

外部取込（一括・1枚ずつの両方）に「ファイル名からタグ・権利者を取得する」モードを追加しました。BorgesTag は保存時にファイル名へタグ・権利者を埋め込む（`-タグ-権利者` 形式）ため、このモードを ON にすると、ファイル名に含まれる文字列を登録済みのタグ一覧・権利者一覧と照合し、一致したものをタグ・権利者として保存履歴へ復元します。保存履歴が失われた場合に、ディスク上の画像ファイル名から元のタグ・権利者を回収する用途を想定しています。

二重付与を防ぐため、このモードが ON のときは取込時の通常のメタ付与（ファイル名へのタグ・権利者の再付与）を行いません。ファイル名はそのまま保ち、含まれていた情報のみを保存履歴側に取り込みます。

照合は登録済みタグ・権利者との完全一致（NFKC 正規化・大文字小文字無視）で行うため、一覧に無い語や `-` を含むタグは取りこぼす場合があります（保存履歴の復旧補助という位置づけ）。

[design-doc-skip: 設計書（画面別/12 外部取込）へ新規 storage key extImportFromFilename とモード UI を反映済み]

## [1.46.33] - 2026-05-29

### Fixed — 取り込み予定リストのフォルダを削除しても 1枚ずつ取り込みセッションが残る不具合

外部取込の取り込み予定フォルダリストでフォルダ行を削除しても、そのフォルダに紐づく「1枚ずつ取り込み」セッション（取り込み待ちの一覧）が残ってしまい、件数表示などに反映され続ける不具合を修正しました。フォルダ行の削除とセッションが別管理で、行を消してもセッションが消えなかったことが原因です。

本リリースから、フォルダ行を削除した際に、そのフォルダ（またはサブフォルダ）に紐づくセッションも自動的に除去されるようにしました。削除の確認ダイアログにも「紐づく取り込みセッションも削除されます（保存済みの履歴は削除されません）」と明記しています。

これにより、古いスキャン結果のセッションが残って取り込み件数が想定と合わなくなる状況を防ぎます。保存済みの履歴データには一切影響しません。

## [1.46.32] - 2026-05-29

### Fixed — 外部取込の日付フィルタ（基準日時・開始日時）が効かず全ファイルが対象になる不具合

外部取込で開始日時・基準日時を指定しても絞り込まれず、フォルダ内の全ファイルが対象になってしまう不具合を修正しました。原因は、拡張機能側が日時を UTC（協定世界時）形式に変換して渡していたため、ネイティブ側で日時の比較に失敗し、フィルタ条件が無視されてファイルの取りこぼし判定が効かなくなっていたことです。

本リリースから、指定した日時をそのままのローカル時刻でネイティブ側へ渡し、ファイルの更新日時（ローカル時刻）と正しく比較するよう修正。これにより、開始日時を指定すれば「その日時以降に保存したファイルだけ」を正しく絞り込めるようになり、特定期間に保存した画像の回収が意図どおり動作します。

ネイティブ側（image_saver.py 1.11.4 → 1.11.5）にも、万一タイムゾーン付きの日時が渡された場合にローカル時刻へ正規化する防御を追加。

[design-doc-skip: 日時フィルタの内部表現（タイムゾーン扱い）の修正のみで、新規 storage key・UI・データ仕様の変更なし。diff 中の extImportCutoffDate / extImportFromDate は既存キー（画面別/12 記載済）]

## [1.46.31] - 2026-05-29

### Fixed — 外部取込で大きいフォルダをスキャンするとネイティブアプリが切断される不具合

画像の大元フォルダなど多数のファイルを含むフォルダを外部取込でスキャンすると、「ネイティブアプリが切断されました」となり取込できない不具合を修正しました。原因は、スキャン結果を 1 つの応答メッセージで返していたため、結果が大きいと Native Messaging のメッセージサイズ上限（約 1MB）を超えて接続が切れていたことです。

本リリースから、スキャン結果が大きい場合はネイティブ側が結果を一時ファイルへ書き出し、拡張機能側が分割読み込みで復元するよう変更（GIF サムネイルで既に使われている方式と同じ）。これにより、フォルダ内のファイル数に関わらずスキャン結果を取りこぼしなく取得できます。日付範囲で絞り込んだ大元フォルダの一括スキャンも切断されずに完了します。

ネイティブ側（image_saver.py 1.11.3 → 1.11.4）。結果が約 800KB 未満の通常のスキャンは従来どおり応答に直接含めるため、挙動・速度に変化はありません。

[design-doc-skip: 本変更は SCAN_EXTERNAL_IMAGES 応答の chunk 化のみで新規 storage key なし。diff 中の extImportCutoffDate / extImportFromDate は v1.46.29 で追加・画面別/12_設定画面_外部取り込みタブ_詳細.md に記載済の既存キー]

## [1.46.29] - 2026-05-29
